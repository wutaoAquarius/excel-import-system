## 一、项目概述

### 1.1 项目背景
物流/快递行业需要频繁批量下单，但客户提供的Excel模板格式各异：
- 列名不同（如"收件人"vs"收货人"vs"收方"vs"Receiver"）
- 列序不同（字段排列顺序各异）
- 表头行数不同（有的模板第1行就是数据，有的有标题行/合并单元格）
- 字段组合不同（有的拆分省市区，有的合并完整地址）
- 缺少选填字段（部分模板不包含外部编码或备注列）

### 1.2 解决方案
开发一个Web应用，**支持多种Excel模板的自动识别与导入**，完成批量下单流程。

### 1.3 核心价值
- 支持至少2种不同模板格式的自动识别
- 模板记忆学习功能
- 支持1000+条数据的无缝导入
- 完整的数据校验与错误提示

## 四、功能模块详细需求

### 模块一：模板管理与文件导入 (46分)

#### 1.1 文件上传
- ✅ 支持上传 `.xlsx` 和 `.xls` 文件
- ✅ 支持拖拽上传和点击上传
- ✅ 文件格式错误时友好提示
- ✅ 空文件检查
- ✅ 编码异常处理

#### 1.2 多模板自动识别（7分）
**目标**: 自动识别不同列名指向同一字段

**示例**:
```
"收件人" / "收货人" / "收方" / "Receiver" → 字段：收件人姓名
"电话" / "联系电话" / "Phone" → 字段：收件人电话
```

**识别策略**:
- 使用相似度匹配（如Levenshtein距离、余弦相似度）
- 支持中英文别名映射
- 同一字段多个列名的优先级处理

#### 1.3 列序适配（6分）
**目标**: 无论列的排列顺序如何，都能正确映射

**示例**:
- 温层在第1列 vs 第10列
- 重量/件数位置互换
- 地址字段分散或集中

**实现方式**: 基于列名识别，而非列位置

#### 1.4 手动映射（4分）
**场景**: 自动识别失败或用户想自定义映射

**功能**:
- 显示"未识别的列"列表
- 提供下拉菜单让用户手动选择对应系统字段
- 允许跳过无关列
- 实时预览映射结果

#### 1.5 模板记忆学习（10分）⭐ **重点难点**
**目标**: 一次手动调整，永久记住此映射规则

**实现流程**:
1. 用户手动调整完列映射关系后，系统自动记录此次映射规则
2. **生成模板指纹**: 基于表头列名（或其排序哈希）生成唯一ID
3. **保存规则**: 在本地存储或数据库中记录
   ```
   {
     templateFingerprint: "hash_of_headers",
     mapping: {
       "column_A": "发件人姓名",
       "column_B": "发件人电话",
       ...
     },
     timestamp: 2026-05-07T...
   }
   ```
4. **下次识别**: 上传新文件时，计算其表头指纹，查询历史映射
5. **自动应用**: 如果找到相同或相似指纹，自动应用映射，无需再次调整

**相似度判断**: 可用集合相似度（Jaccard）或编辑距离

#### 1.6 实时进度条（与导入同步）
- 显示解析进度百分比 (0% - 100%)
- 显示当前处理条数 / 总条数
- 支持取消操作

#### 1.7 大文件处理（4分）
- 支持 1000+ 条数据导入
- 不卡顿、不崩溃
- **建议方案**: 分块处理（WebWorker或后端分页）

#### 1.8 字段缺失容错（3分）
**场景**: 模板缺少选填字段（如无"外部编码"或"备注"列）

**处理方式**:
- 缺少选填字段 → 该字段置空，继续处理
- 缺少必填字段 → 提示用户并拒绝导入
- 支持部分列缺失的模板

#### 2.3 行内实时校验（5分）⭐ **重点难点**
**必检查内容**:
1. **必填校验**: 发件人/收件人信息、重量、件数、温层

#### 2.4 全部错误一次性展示（关键要求）
**需求**: 校验时必须一次性列出**所有错误**，明确标注：
- 具体行数
- 字段名
- 错误原因

**示例格式**:
```
❌ 错误汇总（共5个错误，请修正后再提交）

1. 第3行，收件人电话：格式错误
2. 第5行，重量：不是正数
3. 第8行，发件人地址：缺失必填字段
4. 第12行，外部编码：与第8行重复
5. 第15行，温层：不在可选范围内（常温/冷藏/冷冻）
```

**实现**:
- 不得一次只提示一个错误让用户反复修改（❌ 反面示例）
- 应该一次性显示所有问题

## 五、数据校验规则总结

### 5.1 必填字段校验（4分）
```
必填字段: [发件人姓名, 发件人电话, 发件人地址, 收件人姓名, 
          收件人电话, 收件人地址, 重量, 件数, 温层]

缺失 → 标红 + 错误提示 "XXX：缺失必填字段"
```

### 5.2 格式校验（4分）
```
电话格式: /^1[3-9]\d{9}$/ 或其他合法格式
  错误 → 标红 + 错误提示 "电话格式错误"

重量: 必须是正数 (> 0)
  错误 → 标红 + 错误提示 "重量不是正数"

件数: 必须是正整数 (> 0 且为整数)
  错误 → 标红 + 错误提示 "件数必须是正整数"

温层: 必须在 ['常温', '冷藏', '冷冻'] 之一
  错误 → 标红 + 错误提示 "温层值不在范围内"
```

### 5.3 重复检测（4分）
```
外部编码:
  - 同批次内重复 → 标红 + "第X行，外部编码：与第Y行重复"
  - 与已存在数据重复 → 标红 + "与历史数据重复"
  - 可选：允许留空（外部编码非必填）
```

### 5.4 文件异常处理（3分）
```
空文件 → "文件为空，请检查"
格式错误 → "不支持的文件格式，请上传 .xlsx 或 .xls"
无有效Sheet → "无法找到有效的数据表，请检查文件结构"
编码异常 → "文件编码错误，请使用UTF-8或GBK编码"
```

## 十一、容易忽略的细节（建议自我反思）

### 11.1 关键细节清单
1. **模板记忆学习机制** - 很多人只做自动识别和手动映射，忽略了自动学习
2. **全量错误显示** - 逐条提示错误会严重影响用户体验，必须一次性显示
3. **外部编码重复检测** - 需要同时检查批次内和历史数据，逻辑复杂容易出错
4. **列名相似度匹配** - 使用简单字符串对比会导致识别率低，需用算法
5. **大文件性能** - 1000+ 行数据的虚拟化渲染非常重要
6. **数据库持久化** - 有些人可能只存在本地，导致重新刷新后丢失
7. **导出功能** - 需要保留修改状态，不是原始数据
8. **表头固定** - CSS实现表头固定并支持横向滚动，技术细节多
9. **单元格编辑的键盘交互** - Tab/回车/Esc的完整支持
10. **国际化支持** - 虽未明确要求，但英文列名识别需要考虑

## 十三、常见误区

### 13.1 功能实现误区
- ❌ 只支持固定列序（必须支持任意列序）
- ❌ 手动映射后不保存（必须记忆学习）
- ❌ 逐条显示错误（必须全量显示）

- ❌ 数据存本地（必须持久化到数据库）

- ❌ 忽视UI美观度（占5分）
